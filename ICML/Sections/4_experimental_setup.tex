\section{Experimental Setup}
\label{sec:experiments}

This section describes our experimental setup for training (\S\ref{sec:train_setup}) and evaluation (\S\ref{sec:eval_benchmarks}), and describes baseline approaches we compare against (\S\ref{sec:baselines}).

\subsection{Training Setup}\label{sec:train_setup}

We process the SWE-Smith~\citep{yang2025swesmithscalingdatasoftware} dataset using the procedure described in \S\ref{sec:data_creation}, resulting in a filtered training set of 39K instances drawn from 128 repositories. None of these repositories overlap with those used in our evaluation benchmarks (\S\ref{sec:eval_benchmarks}), avoiding any risk of dataset contamination.

We train models from the Qwen3 family~\citep{yang2025qwen3technicalreport}, namely \texttt{Qwen/Qwen3-1.7B}, \texttt{Qwen/Qwen3-4B-Instruct-2507}, and \texttt{Qwen/Qwen3-14B}. For the 1.7B and 14B models, we disable thinking and adopt a modified chat template that preserves the \texttt{<think>} and \texttt{</think>} tokens from earlier turns during tokenization\footnote{https://huggingface.co/OpenPipe/Qwen3-14B-Instruct}, overriding the default behavior which removed them. This modification enables trajectory-level training by concatenating the agent’s actions and environment observations across the multi-turn interaction into a single sequence, instead of performing computationally expensive step-level training. We mask the loss on tokens not generated by the model, including the system prompt, user prompt, and environment observations (i.e., tool responses). To prevent re-tokenization drift\footnote{https://blog.vllm.ai/2025/10/22/agent-lightning.html}, we extract prompt and response token IDs directly from the vLLM inference engine.

\methodname-4B and \methodname-14B are obtained by training their respective base models with reinforcement learning using our modified GSPO algorithm (\S\ref{sec:train_algo}). For \methodname-4B, we train for 200 steps on 1.6K instances with a batch size of 8 and sample 8 rollouts per instance. Training uses a maximum context length of 40K tokens, the reward function from Equation~\ref{eqn:reward}, and a maximum of 6 turns per episode. For \methodname-14B, we train for 300 steps on 9.6K instances with a batch size of 32 and sample 4 rollouts per instance. We use a maximum context length of 50K tokens, extended with YaRN~\citep{peng2026yarnefficientcontextwindow}, and the reward function from Equation~\ref{eqn:reward} augmented with the auxiliary term $r^{\mathsf{turn}}(\tau, k)$. The agent is allowed a maximum of 4 turns per rollout.\sonicomment{is it worth highlighting that our training runs use just 4-6 turns over baselines that use more?}

Since the base \texttt{Qwen3-1.7B} model exhibits very poor performance (near-zero F1 scores) on our evaluation benchmarks (\S\ref{sec:results}), we do not train \methodname-1.7B with RL from scratch. Instead, we warm-start the model using rejection sampling fine-tuning (RFT)~\citep{yuan2023scalingrelationshiplearningmathematical} on trajectories generated by \methodname-14B. Specifically, we run rollouts with \methodname-14B on a random subset of 7.7K training instances and retain only those achieving perfect localization performance across all three granularities (i.e., F1 score of 1.0 for file, module, and function; see \S\ref{sec:eval_benchmarks}), yielding 4K training examples.

We then use the veRL framework~\citep{Sheng_2025} to perform supervised fine-tuning of the base \texttt{Qwen3-1.7B} model on these successful trajectories for one epoch with a learning rate of $5e^{-5}$, a cosine learning rate scheduler, warmup ratio of 0.1, global batch size of 8, and the AdamW optimizer~\citep{loshchilov2019decoupledweightdecayregularization}. The resulting checkpoint (\methodname-1.7B-RFT) is subsequently trained using RL (\S\ref{sec:train_algo}) for 100 steps on 800 instances (not seen during RFT) with a batch size of 8 and by sampling 8 rollouts per instance. We use a maximum context length of 32K tokens, the reward function from Equation~\ref{eqn:reward}, and allow a maximum of 4 turns per rollout.

Across all RL training runs, we use a constant learning rate of $1e^{-6}$, set \texttt{clip\_ratio\_low} to $3e^{-4}$ and \texttt{clip\_ratio\_high} to $4e^{-4}$, use AdamW optimizer, and sample rollouts using a temperature of 1.0. All experiments are conducted on a single node with 8$\times$H100 GPUs.

\subsection{Evaluation Setup}\label{sec:eval_benchmarks}

We report performance on three evaluation benchmarks: SWE-Bench Lite~\citep{jimenez2024swebenchlanguagemodelsresolve}, which contains 300 instances; SWE-Bench Verified~\footnote{https://openai.com/index/introducing-swe-bench-verified/}, which contains 500 instances; and the Python subset of SWE-Bench Pro~\citep{deng2025swebenchproaiagents}, which contains 266 instances and is substantially more challenging than the other two benchmarks. For all benchmarks, we re-purpose the datasets and extract the ground truth locations using the procedure from \S\ref{sec:data_creation}.

During evaluation of Qwen3 models, as well as during rollout generation from \methodname-14B for performing RFT on \texttt{Qwen3-1.7B} (\S\ref{sec:train_setup}), we follow the generation hyper-parameters recommended by Qwen3 model developers. For instruct models and reasoning models with thinking disabled, we use temperature $=0.7$, top-$k=20$, and top-$p=0.8$. For reasoning models with thinking enabled, we use temperature $=0.6$, top-$p=0.95$, and top-$k=20$. Finally, we set the maximum context length to 132K tokens for all Qwen3 models.

\paragraph{Evaluation Metrics:} Our primary evaluation metric is the instance-wise average of F1 score between the predicted locations and the ground truth locations at three granularities: file, module, and function. In addition, we report average precision and recall for each granularity to compare with baselines that predict a fixed number of locations, often resulting in disparities between precision and recall.

\subsection{Baselines}\label{sec:baselines}

We compare \methodname against several baselines using both open-source and closed-source LLMs: RepoNavigator~\citep{zhang2026toolenoughreinforcementlearning}, RepoSearcher~\citep{ma2025toolintegratedreinforcementlearningrepo}, LocAgent~\citep{chen-etal-2025-locagent}, OrcaLoca~\citep{yu2025orcalocallmagentframework}, CoSIL~\citep{liu2025cosil}, and Agentless~\citep{xia2024agentlessdemystifyingllmbasedsoftware}. All metrics for baselines are taken from prior work, either from the corresponding paper or from subsequent work.

All baselines except RepoNavigator output a ranked list of top-$K$ locations (typically $K$ is fixed to 5), whereas \methodname and RepoNavigator predict a variable number of locations, which is a better design choice as different issues often require edits across arbitrarily different number of locations. For the choice of our benchmarks, many of these baseline approaches often have greater recall than precision. We argue that our design is better because higher precision is more crucial than higher recall for downstream issue resolution - ~\citet{cognition2025swegrep} report that polluting the context of the coding agent is more detrimental than leaving some context out, as the agent is typically only a few searches away to recover any remaining context. Refer to Appendix \ref{appendix:topk_discussion} for additional discussion.

We also evaluate other LLMs with OpenHands-Bash (\S\ref{sec:agent_scaffold}). We benchmark the base variants of \methodname LLMs: \texttt{Qwen3-4B-Instruct-2507}, and the non-thinking versions of \texttt{Qwen3-1.7B} and \texttt{Qwen3-14B}. We include \texttt{Qwen3-32B} with thinking enabled to compare with a larger reasoning model. Finally, we evaluate two closed-source models, GPT-5 and Claude Sonnet 4.5. For open-source LLMs, we use the same prompts as \methodname, specifying a turn limit of 4 (or 6 for 4B LLMs) while allowing up to \textbf{15} turns in the backend to ensure a fair comparison, since base models may require more steps to complete the task. For closed-source models, we limit backend turns to 6 due to cost constraints. Despite specifying the turn limit in the system prompt, GPT-5 and Claude Sonnet 4.5 frequently exhaust all steps without submitting predictions, yielding a score of 0. Following RepoNavigator~\citep{zhang2026toolenoughreinforcementlearning}, we mitigate this by adding a reminder message before the last turn prompting the LLM to submit its final answer.
