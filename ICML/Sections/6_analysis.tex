\subsection{Does Effective Code Localization Improve Issue Resolution?}\label{sec:downstream_swebench}

We demonstrate that augmenting issue-resolution agents with code localized by \methodname improves both downstream performance and efficiency on SWE-Bench Verified when using the OpenHands Software Agent SDK~\citep{wang2025openhandssoftwareagentsdk}. We consider three settings: (1) a vanilla baseline without any localization, (2) augmenting the agent with locations retrieved by \methodname-14B, and (3) augmenting the agent with oracle locations parsed from the gold patch (\S\ref{sec:data_creation}). For settings (2) and (3), we modify the user prompt by including the names of localized files, modules, and functions (see Appendix~\ref{appendix:prompts_swe_bench}).

We perform experiments using two LLMs: Qwen3-4B-Instruct-2507 and Qwen3-Coder-30B-A3B-Instruct, using a 128K context-window, a maximum of 100 turns, and recommended decoding hyperparameters from Qwen3 developers. We compare these three approaches based on performance, using issue resolution rate (\% of issues successfully fixed), and efficiency, using instance-level averages of number of steps per trajectory, and total input and output tokens accumulated over all turns.

We present experimental results in Table \ref{tab:downstream_swe_bench}. For Qwen3-4B-Instruct-2507, improved localization quality leads to substantial gains in both performance and efficiency: augmenting the agent with locations retrieved by \methodname-14B increases the resolution rate by \textbf{3.80\%}, while using \textbf{2.18} steps fewer on average, and reducing input and output tokens by \textbf{17.46\%} and \textbf{6.71\%}, respectively. Similar trends are observed for Qwen3-Coder-30B-A3B-Instruct, where augmentation with \methodname-14B yields notable efficiency gains with \textbf{2.89} fewer steps on average and reductions of \textbf{5.33\%} and 2.00\% in input and output tokens, while achieving a slightly higher resolution rate (+0.80\%). For both models, replacing \methodname localization with oracle locations results in a further performance boost (\textbf{+6.00\%} for 30B and \textbf{+2.40\%} for 4B). The 30B model additionally benefits from improved efficiency, with \textbf{1.37} fewer steps and reductions of \textbf{10.12\%} and \textbf{5.60\%} in input and output tokens. Overall, these results indicate that continued improvements in code localization will likely further enhance both the performance and efficiency of issue resolution agents.

\subsection{How does the tool-use behaviour of \methodname evolve during RL?}\label{sec:tool_use_behaviour}
To study the evolving tool-use behavior of \methodname 4B and 14B during training, we analyze the distribution of Unix utilities invoked by the model. Specifically, we use rollouts logged during training and parse tool calls to the \texttt{Terminal} tool and identify the Unix utilities leveraged by the agent. For analysis, we consider rollouts corresponding to model checkpoints at every 10 training steps. Command usage is aggregated across these checkpoints and the top-8 most frequent utilities are selected. For each of these model checkpoints (at every 10 training steps), we then compute the relative frequency with which these Unix utilities are used across all rollouts logged in that training step, and visualize how these frequencies evolve during training using the heatmaps in Figure~\ref{fig:merged_tool_use}.

Interestingly, we observe that the 14B model eventually converges to using just two Unix utilities: ripgrep (rg) and sed. While the 14B model uses many other utilities like grep, find, wc, cat, etc. during the initial training phases, we observe that after 200 training steps, the 14B model only uses two utilities: rg (mostly for keyword-based code search) and sed (to read particular line ranges in files). Similarly, the 4B model uses more utilities during the initial phases of training but eventually converges to mainly using rg, sed, cat, and xargs. These findings indicate that it is possible to train LLMs for effective code localization using even simpler agent scaffolds which only support a very small subset of Unix utilities, instead of equipping the agent with a bash terminal.
